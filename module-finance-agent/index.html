Phase 1 — Always validate before you do anything expensive

Node 1

Webhook

Listens at /financial-research for incoming requests

Configured in responseNode mode — a downstream node controls the HTTP reply

Node 2

GETtickermap

Fetches the full SEC EDGAR company list — every registered public company and their CIK identifier

Ready and waiting before validation completes

Node 3

Validate ticker

Three rules: reject empty input · enforce 1–5 uppercase letters (NASDAQ format) · block reserved strings like TEST, NULL

Protects downstream nodes from bad input

Node 4

Extract CIK

Looks up the ticker in the company map

Zero-pads the CIK to 10 digits (SEC URL format: CIK0000320193)

Outputs a clean { ticker, cik } object

Key principle: Validate early. The cheapest place to catch a bad input is before you've made four API calls.
Ask the room: "Why validate before you fetch data?" The answer is cost and reliability — every wasted API call is time and money. In a production workflow, input validation is always the first node after the trigger.

Phase 2 — Don't make the caller wait

Respond immediately, then keep working

✗ Synchronous (the wrong way)

Caller sends request
  → Workflow runs
    (60 sec of API calls + LLM)
  → Caller waits...
  → Caller's timeout: 30 sec ✗
  → Connection dropped

✓ Async (this workflow)

Caller sends request
  → Validate ticker (< 1 sec)
  → HTTP 202 Accepted
    { status: "accepted",
      ticker: "AAPL" }
  → Caller unblocked ✓
  → Workflow continues in background
  → HTML brief delivered when ready
Why it matters: Most real-world AI workflows take longer than a browser or API client will wait. Delivery via a separate channel could mean: a Slack message, an email, a webhook callback, or a file on a shared drive.
This is an architectural pattern that appears in almost every production AI system. The caller gets a fast acknowledgement; the actual output arrives asynchronously. Ask: "How do results get back to you in the tools you use today?" — they'll usually describe some version of this pattern already.

Phase 3 — Three sources · One go · Not one at a time

Each source is independent — running them sequentially would triple the wait time for no reason

Branch A

SEC EDGAR

Full submission history — 10-K annual, 10-Q quarterly, 8-K current reports

Most authoritative financial source — direct from the regulator

data.sec.gov/submissions/CIK{cik}.json

Branch B

Yahoo Finance

1 year of daily price data — current price, previous close, 52-week high/low, market cap, currency, exchange

query1.finance.yahoo.com/v8/finance/chart/{ticker}

Branch C

News Search

Latest English-language business news mentioning the ticker — titles, sources, dates, descriptions, URLs

newsdata.io/api/1/latest?q={ticker}&language=en

Total wait time for Phase 3 = time of the slowest single request — not the sum of all three. This pattern is worth replicating anywhere you gather context from multiple independent sources.

Parallel = doing multiple things at once. Merge = waiting until they're all done. Together these two patterns are how you build fast, reliable data-gathering workflows.
The Merge node is the synchronisation point. Ask: "Can you think of a workflow in your work where you'd want to pull from multiple sources simultaneously?"

Never send raw API responses to an LLM

The Combine node curates exactly what the model needs to see

Source Raw response What's extracted
SEC EDGAR Full submission history — potentially thousands of filings 3 most recent 10-K, 10-Q, and 8-K only
Yahoo Finance Full year of daily OHLCV price data Current price, prev close, 52-wk high/low, market cap, currency
News Search All articles returned by the API Top 5 articles: title, source, date, description, URL

Context window limit

You can't send 1,000 SEC filings — the LLM simply won't fit them

Signal vs noise

Irrelevant data distracts the model and degrades output quality

Cost

Smaller prompts = fewer tokens = lower cost per workflow run

This is a direct callback to Session 1 — context windows are finite. The Combine node is doing the work of a human analyst who'd scan through hundreds of filings and pick only the relevant ones before writing the brief. Curating the context before the LLM sees it is one of the most impactful things you can do for output quality.

Building the Prompt — The most important node in the workflow

System Prompt

Role

You are a senior equity research analyst.

Instruction

Generate a structured, factual financial research brief based strictly on the data provided.

Constraint

Do not speculate, hallucinate figures, or make buy/sell recommendations.

Fallback

Mark missing sections [DATA UNAVAILABLE].

User Message + Output Format

Generate a financial research brief for {ticker}. SEC FILINGS: {10-K, 10-Q, 8-K metadata} FINANCIALS: {price, market cap, 52-wk range} RECENT NEWS: {top 5 news articles} Structure output as: ## 1. Executive Summary ## 2. Recent SEC Filings ## 3. Financial Highlights ## 4. Recent News & Events ## 5. Risk Flags ## 6. KPI Snapshot
Output

Six ## headings — so the HTML formatter can reliably parse and style each section

This is Session 1 prompt engineering applied directly in production. Point out each element: Role, Instruction, Constraints, Output format. The output format instruction — the six ## headings — is particularly important. By specifying exactly what headings the LLM should produce, the downstream HTML formatter can reliably parse and style the output. Structure in = structure out.

Temperature 0.2 — and why it matters here

Node configuration

ParameterValue
Modelgpt-4o — complex synthesis task
max_tokens2,000 — thorough brief, no runaway generation
temperature0.2 — factual, deterministic, consistent
Running this workflow 100 times on the same ticker produces structurally identical briefs — varying only with the input data. That predictability is exactly what you want for a professional output.

Temperature scale

Temperature is one of the most consequential prompt-level decisions in a production workflow. Ask the room: "Would you want the model to be creative when generating a financial brief?" — the answer is clearly no. But you might want creativity for brainstorming workflows. They now have the mental model to make that call themselves.

From LLM markdown to a shareable HTML document

The Format as HTML node converts the raw LLM response into a styled, publication-quality file

Input (raw LLM text)

## 1. Executive Summary Apple Inc. (AAPL) reported... **Key financials:** - Revenue: $94.9B (Q1 FY2025) - Market cap: $3.2T
↓ JavaScript code node
<h2>1. Executive Summary</h2> <p>Apple Inc. (AAPL) reported...</p> <p><strong>Key financials:</strong></p> <ul> <li>Revenue: $94.9B (Q1 FY2025)</li> <li>Market cap: $3.2T</li> </ul>

Output file

Georgia serif font — equity research house style

860px container — readable at screen width

Disclaimer footer — important for any financial output

Named AAPL_brief_2025-03-25.html — encoded as Base64 for download

Reusable pattern: LLM → raw text → code node → formatted document. Works for reports, proposals, briefings, summaries.
Point out that the LLM was instructed to use ## headings precisely so this formatter can reliably find and convert them. The workflow is a chain where each node's output is designed to be the next node's input. That design thinking is the core skill.

Five patterns to take everywhere

Every high-quality AI workflow you build will use some combination of these

1

Validate at the entry point

Bad input causes cascading failures. The cheapest fix is at the front door.

2

Respond asynchronously

Don't make callers wait for long-running AI tasks. Acknowledge immediately, deliver when ready.

3

Gather context in parallel

Independent data sources should run simultaneously. Use a Merge node to synchronise.

4

Normalise before the LLM

Curate what the model sees. Less noise = better output. Structured input = structured output.

5

Match temperature to the task

Low temperature for factual, consistent outputs. Higher temperature for creative tasks.

These are the five things to write down. Everything else in the workflow is implementation detail — these patterns are architectural principles that apply to any domain: HR, operations, procurement, product, legal. Ask: "Which of these five would be most useful in the workflow you sketched at the end of Session 4?"
