Chatbot vs AI Agent

A chatbot answers. An agent acts.

Chatbot (what most people use today)
  • You send a message → it replies → done
  • Each interaction is independent
  • You drive every step
"Summarize this email" → you get a summary → you copy it somewhere manually
AI Agent
  • You give it a goal → it figures out the steps → it executes
  • Can take multiple actions in sequence
  • Handles decisions along the way
"Monitor my inbox, summarize project updates, add to my tracker — every morning"
The key difference: A chatbot is reactive. An agent is proactive and autonomous.
This is the mental model shift of the whole session. Most people's mental model of AI is "I ask, it answers." An agent flips that — you define the goal once, it does the work repeatedly. Use the inbox example and ask: "How long does that task take you manually each morning?"

Every agent is built from the same 5 parts

ComponentWhat it isExample
Signal What triggers the agent to start A new email arrives, a form is submitted, a scheduled time
Input The data the agent works with The email content, a document, a spreadsheet row
Reasoning (LLM) The "brain" — plans and decides Claude, GPT-4o — reads input, decides next action
Actions / Tools What the agent can do Search the web, read a file, send an email, update a spreadsheet
Memory What the agent remembers Previous interactions, user preferences, accumulated context

Signal: What Starts an Agent

Agents don't wait — they respond to triggers

Signal TypeExample
MessageSomeone sends a message to the agent directly
ScheduleRuns every morning at 8am automatically
EventA form is submitted, a file is uploaded to SharePoint
Data changeA new row is added to a spreadsheet or tracker
API / webhookAnother system sends a notification to trigger the agent
The signal removes the human trigger. The agent sits on standby and activates when the right condition is met — this is what makes it feel autonomous.
Give a vivid example: "Imagine an agent that triggers every time a new project is added to your tracking sheet — it automatically creates the folder structure, drafts the kickoff email, and schedules the first meeting." That's a data-change signal driving a sequence of actions, with zero manual steps.

Actions & Tools: What an Agent Can Do

The agent is only as powerful as its tools

ToolWhat it enables
Web searchLook up live information from the internet
File readerRead documents, spreadsheets, PDFs
Email / calendarSend emails, create meetings, read inbox
Spreadsheet writeUpdate or add rows to a tracker or data file
Form submissionFill and submit web forms automatically
API callsTalk to external systems — CRM, ERP, databases
Code executionRun calculations, process and transform data
Key insight: The LLM itself can't do anything — it can only decide what to do. Tools are what give the agent real-world capabilities.
The LLM is the brain; the tools are the hands. Without tools, an agent is just a very smart planner with no ability to act. This also explains why different agents have different capabilities — it depends entirely on what tools they've been given access to.

Memory: What an Agent Remembers

Memory is what turns a task into a relationship

Short-Term Memory (in-session)

  • Everything within a single conversation or workflow run
  • Automatically available — the context window
  • Lost when the session ends

Long-Term Memory (persisted)

  • Stored externally — a database, a file, a notes system
  • Explicitly saved and retrieved between sessions
  • Survives across sessions and interactions
  • "This client prefers bullet-point summaries" → stored, retrieved next time
Without memory: every interaction starts from scratch. With memory: the agent gets smarter and more personalised over time.
Draw the analogy to a new colleague vs. a trusted one who knows your preferences. Then ask: "What would you want your AI assistant to remember about how you work?" Most consumer tools (Copilot, ChatGPT) now offer optional memory — this isn't theoretical.

Agent Architectures

How agents are structured to handle complex work

Pattern 1

Sequential (Pipeline)

A → B → C → Output

Each step feeds into the next in a fixed sequence

Example: Receive brief → research → draft → format → send
Best for: predictable, linear processes
Pattern 2

Hierarchical

Manager ├─ Sub-agent A ├─ Sub-agent B └─ Sub-agent C

One "manager" agent coordinates specialist sub-agents

Example: Research + Writing + Review agents coordinated by a planner
Best for: complex tasks with distinct specialisms
Pattern 3

Multi-agent (Collaborative)

Draft ⟷ Critique ⟷ Fact-check

Multiple agents work in parallel, share findings, check each other

Example: One drafts, one critiques, one fact-checks — simultaneously
Best for: high-stakes output needing multiple perspectives
For most practical day-to-day use cases, sequential is what people encounter first. Hierarchical and multi-agent are more relevant for complex automated pipelines. Understanding the concept prepares them for what's possible.

Designing an Agent: The Prompting System

A well-designed agent starts with a well-designed system prompt

ElementWhat to defineExample
Role Who is this agent? "You are a project coordination assistant."
Goal What must it achieve? "Your job is to process incoming project requests and produce a structured brief."
Tools What can it use? "You have access to: web search, email, and the project tracker spreadsheet."
Constraints What must it not do? "Do not make decisions without confirming with the user. Never email external parties."
Output How should it respond? "Always return a structured brief with: Summary, Key Actions, Owner, Deadline."
Think of it as a job description for your agent. A vague job description produces a vague employee. Spend the most time on constraints — that's where most people underinvest.
A well-defined constraint set is the difference between an agent you trust and one that occasionally surprises you. Constraints are not about limiting the agent — they're about making it reliably safe and predictable.

Workflow Design Patterns

The building blocks of any agent workflow

Linear
Trigger → Step 1 → Step 2 → Step 3 → Output
Email received → extract key info → draft reply → send for approval
Branching
Trigger → Decision ├─ If urgent → escalate now └─ If routine → add to queue
New request arrives → Is it urgent? → Yes: escalate immediately / No: add to processing queue
Loop / Retry
Step → Check quality → if not good → retry
Draft a summary → Is it under 100 words? → If not, compress and try again
Most real-world agent workflows combine all three patterns. Start with linear — easiest to build and debug. Add branching when you need to handle different scenarios. Loops are useful for quality-control steps.

Demo 2: Configuring a Custom Agent

Scenario Build an agent that processes incoming project requests and returns a structured brief — consistently, from any format of input.
System Prompt (the agent's instructions)
Agent configuration
You are a project intake assistant. Your goal is to process incoming project requests and produce a structured project brief. When given a project request (in any format — email, bullet points, paragraph), you will: 1. Extract: project name, requestor, objective, key deliverables, deadline, and stakeholders 2. Identify any missing information and list it as open questions 3. Return a structured brief using this format: Project Name: Requestor: Objective: Key Deliverables: Deadline: Stakeholders: Open Questions: Do not make assumptions about missing information. If something is unclear, list it as an open question.
Test with a real, unstructured request
Input sent to the agent
Hey, can someone look into setting up a new reporting dashboard for the ops team? Sarah asked for it, needs to be ready by end of Q2, should show daily order volumes and fulfilment rates. Not sure who else is involved.
Agent Output
Notice: It extracted structured output from a casual, unformatted message — and identified 6 missing pieces of information as open questions, without being told to. This is reusable: same agent, any incoming request, consistent output every time.
This is the core demo of the session. The agent extracted structure from chaos and surfaced missing information. Emphasise: this is just a prompt — no code, no special platform required. The same system prompt in any LLM tool produces this consistently.

Your Turn — Design Your Agent

5 minutes · Think of a task you do regularly

Write a System Prompt for Your Agent
You are a [role]. Your goal is to [what it must achieve]. You have access to: [tools — even if hypothetical for now]. When given [type of input], you will: 1. [Step 1] 2. [Step 2] 3. [Step 3] Always return [output format]. Never [key constraint].
Debrief question: What task did you pick — and what would this agent save you each week?
Walk around and help. Common stumbling blocks: people pick tasks that are too vague ("help me with emails") — push them to be specific about what type of email and what output format. The constraint field is often left empty — prompt them to add at least one. Even a hypothetical agent design is valuable practice for thinking in systems.
